# WebRTC: групповые звонки с общим полотном

Демо групповых видеозвонков в браузере (до 8 участников) с необычным UI: все камеры и
демонстрации экранов — это плитки на **общем полотне**. Любой участник может двигать
плитки, менять их размер, порядок слоёв и разворачивать на всё полотно — раскладка
синхронизируется, и все видят одну и ту же композицию.

Без фреймворков и сборки: ванильный JS на клиенте, Node.js + [`ws`](https://github.com/websockets/ws) на сервере.

## Быстрый старт

```bash
npm install
npm start          # http://localhost:3000
```

Откройте страницу в двух вкладках, введите одинаковый ID комнаты — звонок соединится.

Для устройств вне localhost нужен HTTPS (требование `getUserMedia`). Простейший путь — [Tailscale](https://tailscale.com):

```bash
tailscale serve --bg 3000
# выдаст https://<машина>.<tailnet>.ts.net с автоматическим TLS-сертификатом
```

Бонус: участники в одном tailnet соединяются напрямую даже за строгими NAT.

## Возможности

- **До 8 участников** в комнате, видео + звук.
- **Демонстрация экрана** — у каждого участника, все одновременно. Со звуком вкладки
  (Chrome: при шаринге вкладки; весь экран со звуком — только Windows).
- **Общее полотно**: перетаскивание плиток, ресайз за правый нижний угол,
  слои ▲/▼, разворот плитки на всё полотно ⛶, полотно во весь экран.
  Раскладку меняет любой участник — синхронизация в реальном времени (при
  перетаскивании — 20 обновлений/сек).
- **Сайдбар** со списком участников и экранов; клик по строке поднимает плитку наверх.
- Выключение микрофона/камеры, обработка «камера занята» (звонок продолжится без видео).

## Архитектура

```
Браузер A ─┐                    ┌─ Браузер B
           ├── WebSocket ───────┤
Браузер C ─┘   (сигналинг,      └─ Браузер D
               раскладка)
     │              │
     └── RTCPeerConnection (медиа, P2P-меш: каждый с каждым) ──┘
```

### Сервер (`server.js`)

Один файл, две роли:

1. **Статика** — раздаёт `public/` (с `Cache-Control: no-cache`, потому что клиент и
   протокол сигналинга меняются вместе — устаревший закэшированный клиент не сможет
   говорить с сервером).
2. **Сигналинг** — WebSocket-релей. Комната — это `Map<peerId, WebSocket>`. Сервер
   **не знает ничего про WebRTC**: он выдаёт числовые id, сообщает о входе/выходе
   участников и пересылает JSON-сообщения:
   - с полем `to` — конкретному участнику (SDP-описания, ICE-кандидаты);
   - без `to` — всем в комнате (раскладка, карта стримов).

   Поле `from` сервер проставляет сам — подделать отправителя нельзя.

### Медиа: полный меш

Каждый участник держит по отдельному `RTCPeerConnection` на каждого другого.
Для N участников это N−1 исходящих копий видео — поэтому лимит 8; дальше нужен
SFU (медиасервер), что выходит за рамки демо.

Каждая пара соединений договаривается по паттерну **perfect negotiation**
([MDN](https://developer.mozilla.org/en-US/docs/Web/API/WebRTC_API/Perfect_negotiation)):
обе стороны могут инициировать ренегоциацию (добавили/убрали экран), а коллизии
offer'ов разрешаются ролями «вежливый/невежливый». Роль детерминирована: уступает
тот, у кого меньше id. ICE-кандидаты, пришедшие до `setRemoteDescription`,
буферизуются.

### Как отличаются камера и экран

Камера+микрофон и экран отправляются **разными `MediaStream`**. Идентификаторы
стримов переживают SDP-переговоры, поэтому каждый участник рассылает карту
`{streamId: 'cam' | 'screen'}` (сообщение `stream-meta`). Получатель по этой карте
раскладывает входящие стримы по плиткам `<id>-cam` / `<id>-screen`. Когда шаринг
останавливается, из карты пропадает `screen` — плитка удаляется у всех.

### Общее полотно

Раскладка — простой словарь:

```js
layout = {
  '3-cam':    { x: 0.04, y: 0.08, w: 0.28, z: 13, max: false },
  '3-screen': { x: 0.10, y: 0.05, w: 0.80, z: 1,  max: false },
  ...
}
```

Координаты **нормализованы** (доли размера полотна), поэтому полотна разных размеров
у разных участников показывают одинаковую композицию; высота плитки следует за
аспектом видео. `z` — порядок слоёв; после каждого ▲/▼ значения пересчитываются в
компактные `1..N` — отрицательный `z-index` уводил бы плитку за фон полотна.
`max: true` разворачивает плитку на всё полотно.

Изменения рассылаются сообщениями `layout` (одна плитка, при перетаскивании — с
троттлингом 50 мс) и `layout-all` (весь словарь — после смены слоёв и новичку при входе).
Конфликты решаются просто: последнее пришедшее выигрывает — для демо достаточно.

### Поток событий при входе третьего участника

```
C → сервер: join
сервер → C: joined {id: 3, peers: [1, 2]}
сервер → A, B: peer-joined {id: 3}
C: создаёт RTCPeerConnection к 1 и к 2, рассылает stream-meta
A, B: создают RTCPeerConnection к 3, шлют stream-meta и layout-all
пары (1,3) и (2,3): обмен description/candidate через сервер → P2P-медиа
```

## Структура

```
server.js          — статика + WebSocket-сигналинг (~120 строк)
public/
  index.html       — разметка: вход, полотно, сайдбар, кнопки
  main.js          — вся клиентская логика: меш, плитки, раскладка, шаринг
  style.css        — тёмная тема, плитки, сайдбар
```

## Ограничения

- **Только STUN** (Google), TURN-сервера нет: участники за симметричными NAT в разных
  сетях могут не соединиться. Решения: общий tailnet (см. выше) или свой
  [coturn](https://github.com/coturn/coturn) в `RTC_CONFIG` в `main.js`.
- Меш ≤ 8 участников; для больших комнат нужен SFU (mediasoup, LiveKit, Janus).
- Комнаты без паролей, сигналинг без авторизации — это демо.
- iOS Safari не поддерживает `getDisplayMedia` — кнопка шаринга там скрыта.

## Тесты

E2E-сценарий на `puppeteer-core` (три headless Chrome с фейковыми камерами:
вход, взаимная видимость видео, синхронизация слоёв, выход участника) — запускался
в ходе разработки; скрипт лежит вне репозитория.
